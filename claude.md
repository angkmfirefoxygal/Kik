# claude.md

## 프로젝트 개요: TOPIK 쓰기 피드백 MVP

이 프로젝트는 다음 가설을 검증하기 위한 최소 기능 MVP이다:  
“한국어 학습자들은 TOPIK 쓰기를 가장 어려워하며, 특히 TOPIK II 53·54번 문제에서 높은 부담을 느낀다.”

이 MVP의 1차 목표는 **완성도 높은 서비스 구축**이 아니라,  
**실제 학습자 반응·도움이 되는지 여부·지불 의향**을 확인하는 것이다.

---

## 핵심 사용자 흐름 (User Flow)

1. **TOPIK II 실제 기출 문제(53/54 중 1~2개)를 화면에 표시한다.**
   - 문제 지문
   - 요구 조건(예: 600~700자)
   - 간단한 안내 문구(예: “아래에 한국어로 답안을 작성해 주세요.”)

2. **학습자가 텍스트 입력창에 답안을 작성한다.**
   - 하나의 큰 `textarea` 또는 에디터
   - 한국어 입력에 최적화된 기본 스타일

3. **실시간 글자 수 카운터를 제공한다.**
   - 예: `현재 354자 / 목표 600~700자`
   - 목표 범위를 넘거나 부족하면 색상만 살짝 바꿔주는 정도의 단순 피드백

4. **“피드백 받기” 버튼 클릭 시 LLM API를 호출한다.**
   - 프론트엔드에서 문제 텍스트 + 학습자 답안을 함께 전송
   - LLM은 다음 항목별로 평가:
     - 내용(과제 수행 여부, 논지, 예시 활용 등)
     - 구조(도입–전개–결론, 문단 구성 등)
     - 문법/어휘(문법 오류, 어휘 적절성 등)
     - 표현/연결성(연결어 사용, 문장 간 자연스러운 흐름 등)

5. **LLM 평가 결과를 구조화된 형태로 화면에 표시한다.**
   - 항목별 점수(1~5점)
   - 항목별 한두 줄 피드백
   - 예시:
     - 내용: 3/5 – 과제는 수행했지만 예시가 부족합니다.  
     - 구조: 2/5 – 도입과 결론이 약하고, 본론이 한쪽 주장에만 치우쳐 있습니다.  
     - 문법/어휘: 4/5 – 전반적으로 좋지만 조사 실수가 몇 군데 보입니다.  
     - 표현/연결성: 3/5 – 그러나/반면에 등 연결어를 좀 더 다양하게 써보면 좋겠습니다.

6. **간단한 만족도 질문(선택)**
   - “이 피드백이 도움이 되었나요?”  
     - 예 / 아니오 버튼
   - 초기 MVP 단계에서는 단순히 UI 상의 선택만 처리하거나, 아주 간단한 로그 정도만 남겨도 충분하다.

## 프론트엔드 기술 요구사항

### 기술 스택

- **프레임워크**
  - React + Vite + TypeScript

- **스타일링**
  - Tailwind CSS
  - 목표: 빠르게 프로토타입을 만들 수 있을 정도의 최소 스타일
  - 반응형은 “깨지지 않는 정도” 수준이면 충분

---

### 주요 컴포넌트 설계

1. **`ProblemView`**
   - 역할:
     - TOPIK II 53/54번 문제 텍스트를 보여준다.
     - 권장 글자 수(예: 600~700자)를 표시한다.
   - 표시 요소:
     - 문제 제목(예: “TOPIK II 쓰기 54번”)
     - 문제 지문 전체
     - “권장 글자 수: 600~700자” 텍스트

2. **`AnswerEditor`**
   - 역할:
     - 학습자가 답안을 작성하는 입력 영역
     - 글자 수를 실시간으로 보여준다.
   - 기능:
     - `textarea` 또는 `contenteditable` 기반 입력
     - 실시간 글자 수 계산
       - 예: `현재 354자 / 목표 600~700자`
     - 글자 수 범위에 따른 단순 시각적 피드백
       - 너무 적으면: 회색/주황색
       - 적정 범위: 기본/초록색(선택)

3. **`FeedbackPanel`**
   - 역할:
     - LLM으로부터 받은 평가 결과를 보여주는 패널
   - 표시 내용:
     - 각 항목별 점수(1~5점)
     - 각 항목별 한두 줄 피드백
   - 상태:
     - 아직 평가 전일 때: “아직 피드백이 없습니다. 답안을 작성하고 ‘피드백 받기’를 눌러주세요.”
     - 평가 요청 중: 간단한 로딩 표시
     - 평가 완료: 점수 + 코멘트 표시

4. **`SatisfactionToggle` (선택)**
   - 역할:
     - 사용자가 피드백 만족도를 간단히 표시
   - UI:
     - “이 피드백이 도움이 되었나요?”
     - [예] [아니오] 버튼
   - 초기 버전에서는 결과를 단순 상태로만 관리해도 된다.

5. **상위 페이지 컴포넌트 (`App`)**
   - 전체 페이지 레이아웃 담당
   - 대략적인 구조:
     - 상단: 간단한 제목/설명
       - 예: “TOPIK 쓰기 연습 + AI 피드백 (MVP)”
     - 좌측/상단: `ProblemView`
     - 중앙: `AnswerEditor`
     - 하단/우측: “피드백 받기” 버튼 + `FeedbackPanel` + `SatisfactionToggle`

### UX/UI 최소 가이드

- 한 화면에서 모든 작업이 가능해야 한다.
  - 문제 보기 → 답안 작성 → 피드백 확인까지 하나의 페이지에서 처리
- 복잡한 네비게이션, 메뉴, 다국어 전환 등은 모두 제외
- 텍스트 위주의 단순한 구성:
  - 흰 배경
  - 카드 형태의 박스 2~3개
  - 기본 폰트, 기본 버튼 스타일에서 약간의 Tailwind 커스터마이징 정도

---

### 백엔드/서버 관련

- 이 문서에서는 **백엔드/서버리스 API 구현 상세는 다루지 않는다.**
- 프론트엔드는 LLM 호출을 위한 API 엔드포인트가 존재한다고 가정하고,
  - "요청 형식"과 "응답 형식(JSON 구조)"에 맞춰 연동만 수행한다.
